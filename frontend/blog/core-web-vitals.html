What Are Core Web Vitals?

Core Web Vitals are a set of user-centered metrics introduced by Google to measure key aspects of user experience on the web. These metrics have become official ranking factors in Google's search algorithm, making them essential for anyone serious about SEO.

The three Core Web Vitals focus on different aspects of the user experience: loading performance, interactivity, and visual stability. By optimizing for these metrics, you not only improve your search rankings but also provide a better experience for your visitors.

Why Core Web Vitals Matter

  • Direct Google ranking factor since 2021
  • Improves user experience and conversion rates
  • Reduces bounce rates and increases engagement
  • Essential for mobile-first indexing
  • Impacts Page Experience signals

The Three Core Metrics

Loading

Largest Contentful Paint (LCP) - measures loading performance

< 2.5 seconds

Interactivity

Interaction to Next Paint (INP) - measures responsiveness

< 200 milliseconds

Visual Stability

Cumulative Layout Shift (CLS) - measures visual stability

< 0.1

Largest Contentful Paint (LCP)

Largest Contentful Paint measures the loading performance of your page's main content. Specifically, it reports the render time of the largest content element visible in the viewport.

What LCP Measures

LCP considers several elements when determining the "largest" content:

  • Images - Including poster images for videos
  • Background images - Applied via CSS
  • Video elements - The poster frame or first frame
  • Block-level elements - Containing text nodes or other inline elements

LCP Thresholds

Rating Time Status
Good Up to 2.5s Pass
Needs Improvement 2.5s - 4.0s Warn
Poor Over 4.0s Fail

How to Improve LCP

  • 1
    Eliminate render-blocking resources

    Minimize CSS and JavaScript that blocks page rendering. Use async/defer attributes and critical CSS techniques.

  • 2
    Optimize images and videos

    Compress images, use modern formats (WebP, AVIF), implement lazy loading, and serve responsive images with srcset.

  • 3
    Use a CDN

    Distribute your content geographically to reduce latency. Cloudflare, AWS CloudFront, and Fastly are popular options.

  • 4
    Preload important resources

    Use <link rel="preload"> for critical fonts, images, and scripts that impact LCP.

  • 5
    Improve server response time

    Aim for Time to First Byte (TTFB) under 600ms. Use faster hosting, enable server-side caching, and optimize database queries.

  • 6
    Enable compression

    Use Gzip or Brotli compression to reduce file sizes transferred over the network.

Preload Example 

<!-- Preload critical image -->
<link rel="preload"
      as="image"
      href="/hero-image.webp"
      fetchpriority="high">

<!-- Preload critical font -->
<link rel="preload"
      as="font"
      href="/fonts/main.woff2"
      type="font/woff2"
      crossorigin>

First Input Delay (FID) & Interaction to Next Paint (INP)

Important update: As of March 2024, Google has replaced FID with a new metric called Interaction to Next Paint (INP) as a Core Web Vital. INP provides a more comprehensive measure of interactivity.

Important Transition Notice

FID is being phased out as a Core Web Vital. INP is now the official metric for measuring interactivity. This section covers both metrics to help with your transition.

What is INP?

Interaction to Next Paint measures the latency of all user interactions throughout the page's lifecycle. It captures the time from when a user interacts with your page (clicks, taps, or key presses) to when the browser can paint the next frame in response.

INP vs FID

FID (Being Replaced)

  • • Only measured first interaction
  • • Didn't account for later interactions
  • • Limited accuracy for overall UX
  • • Only measured input delay

INP (New Standard)

  • • Measures all interactions
  • • Accounts for entire page session
  • • Better reflects real UX
  • • Includes input delay + processing + presentation

INP Thresholds

Rating Time Status
Good Up to 200ms Pass
Needs Improvement 200ms - 500ms Warn
Poor Over 500ms Fail

How to Improve INP

Reduce JavaScript execution time

Break up long tasks using setTimeout, requestIdleCallback, or scheduling APIs. Code-split your JavaScript to reduce initial payload.

Minimize main thread work

Offload work to Web Workers for heavy computations. Use efficient algorithms and avoid unnecessary DOM manipulations.

Use passive event listeners

Add { passive: true } to touch and wheel event listeners to improve scrolling performance.

Optimize event handlers

Debounce or throttle frequently firing events like scroll, resize, and input. Remove event listeners when no longer needed.

Code Example: Debounce Function 

// Debounce function to limit event handling frequency
function debounce(func, wait) {
    let timeout;
    return function executedFunction(...args) {
        const later = () => {
            clearTimeout(timeout);
            func(...args);
        };
        clearTimeout(timeout);
        timeout = setTimeout(later, wait);
    };
}

// Usage
const handleInput = debounce((e) => {
    // Expensive operation like API call
    console.log(e.target.value);
}, 300);

document.getElementById('search')
    .addEventListener('input', handleInput);

Cumulative Layout Shift (CLS)

Cumulative Layout Shift measures visual stability. It quantifies how much visible content shifts unexpectedly during page load. A low CLS ensures a smooth, stable user experience.

What Causes CLS

Layout shifts occur when visible elements change position without user interaction. Common culprits include:

  • Images without dimensions
  • Ads, embeds, and iframes without dimensions
  • Dynamically injected content
  • Web fonts causing FOIT/FOUT
  • Network requests completing late and pushing content down

CLS Thresholds

Rating Score Status
Good Up to 0.1 Pass
Needs Improvement 0.1 - 0.25 Warn
Poor Over 0.25 Fail

How to Improve CLS

Always include width and height on images

Set explicit dimensions on images so the browser can reserve space before they load.

<img src="image.jpg" width="800" height="600" alt="...">

Reserve space for ads and embeds

Use CSS aspect-ratio boxes to reserve space for dynamic content like ads, videos, and iframes.

Preload fonts

Use font-display: swap or optional to prevent invisible text while fonts load.

Avoid inserting content above existing content

If you must insert content, push content down from the top rather than inserting it mid-page.

CSS Aspect Ratio Example 

/* Reserve space for video or responsive image */
.video-container {
    width: 100%;
    aspect-ratio: 16 / 9;
    background: #000;
}

/* Reserve space for dynamic ad */
.ad-slot {
    width: 300px;
    aspect-ratio: 3 / 2;
    background: #f0f0f0;
}

/* Before aspect-ratio was available */
.image-wrapper {
    position: relative;
    width: 100%;
    padding-bottom: 56.25%; /* 16:9 ratio */
}
.image-wrapper img {
    position: absolute;
    top: 0;
    left: 0;
    width: 100%;
    height: 100%;
    object-fit: cover;
}

How to Measure Core Web Vitals

Measuring Core Web Vitals requires both lab testing tools and real-user monitoring (RUM). Each has its place in your optimization strategy.

Lab Data vs Field Data

Lab Data

Controlled testing environment

  • ✓ Consistent results
  • ✓ Easy debugging
  • ✓ Pre-deployment testing
  • ✗ Doesn't reflect real users
  • ✗ Limited network conditions

Field Data (RUM)

Real user monitoring

  • ✓ Real-world conditions
  • ✓ Actual user devices
  • ✓ Google uses for rankings
  • ✗ Variable results
  • ✗ Requires user traffic

Essential Measurement Tools

Google PageSpeed Insights

Provides both lab data (Lighthouse) and field data (Chrome User Experience Report). Enter your URL to get a comprehensive Core Web Vitals report.

pagespeed.web.dev

Lighthouse

Built into Chrome DevTools. Run audits on any page for detailed performance metrics and optimization suggestions.

Search Console

The "Core Web Vitals" report shows actual user data from your site. Grouped by URL group, mobile, and desktop status.

search.google.com/search-console

Web Vitals JavaScript Library

Add to your site to measure Core Web Vitals for real users. Works with any analytics provider.

<script type="module">
import {getCLS, getFID, getFCP, getLCP, getTTFB} from 'https://unpkg.com/web-vitals?module';

getCLS(console.log);
getFID(console.log);
getFCP(console.log);
getLCP(console.log);
getTTFB(console.log);
</script>

Comprehensive Optimization Strategy

Improving Core Web Vitals requires a systematic approach. Follow this roadmap to systematically optimize your website.

Phase 1: Assessment

  • • Run PageSpeed Insights on all key pages
  • • Check Search Console for URL groups with issues
  • • Identify pages with "Poor" ratings
  • • Prioritize high-traffic pages

Phase 2: Quick Wins

  • • Add width/height to all images
  • • Enable compression on your server
  • • Minify CSS and JavaScript
  • • Implement lazy loading for below-fold images

Phase 3: Deep Optimization

  • • Optimize and convert images to WebP
  • • Implement critical CSS
  • • Defer non-critical JavaScript
  • • Reduce server response time

Phase 4: Advanced Improvements

  • • Implement resource hints (preload, prefetch)
  • • Set up a CDN
  • • Optimize third-party scripts
  • • Consider server-side rendering for JavaScript frameworks

Phase 5: Monitoring

  • • Set up Web Vitals JS library
  • • Track metrics in analytics
  • • Set up alerts for regressions
  • • Test changes before deployment

Common Questions & FAQ

Do I need perfect scores on all Core Web Vitals?

No, "Good" ratings are sufficient. Google uses these as signals among hundreds of others. Focus on getting into the "Good" range rather than perfecting scores.

How often does Google update Core Web Vitals?

Google updates the data continuously. The Search Console report shows a rolling 28-day window of data. Changes you make will be reflected as new data comes in.

What if I have thousands of URLs with issues?

Focus on templates and common patterns. Fix the underlying issues affecting multiple pages (like image optimization or CSS loading) rather than individual URLs.

Do Core Web Vitals affect all pages equally?

Core Web Vitals are more important for pages that compete in organic search. Authentication pages, internal tools, and non-public content don't need optimization.

Can third-party scripts affect Core Web Vitals?

Yes, significantly. Third-party scripts for analytics, ads, chat widgets, and social sharing can dramatically impact LCP, INP, and CLS. Load them asynchronously and defer non-critical scripts.
